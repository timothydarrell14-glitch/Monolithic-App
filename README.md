## 🎯 What This Project Is

A simple **Book API** — you can list books, fetch one book, create, update, and delete books. In earlier sessions the data lived in a plain Python list (which vanished every time the server restarted 😅). In this session we upgraded to a **real SQLite database**, managed through an **ORM**, with schema changes tracked by **migrations**.

## 🧠 Core Concepts Explained

### 1. What is an ORM?

**ORM** stands for **Object Relational Mapper**. It's a translator between two worlds:

| The Python world (objects)      | The database world (tables)  |
|---------------------------------|------------------------------|
| A **class** (`Book`)            | A **table** (`books`)        |
| A **class attribute** (`title`) | A **column** (`title`)       |
| An **instance** (`Book(...)`)   | A **row** in the table       |

Without an ORM, you'd write raw SQL strings everywhere:

```python
cursor.execute("INSERT INTO books (title, author) VALUES (?, ?)", ("Dune", "Frank Herbert"))
```

With an ORM, you work with normal Python objects and the SQL is generated for you:

```python
book = Book(title="Dune", author="Frank Herbert")
db.session.add(book)
db.session.commit()
```

**Why is this better?**

- ✅ **Fewer errors** — no hand-written SQL strings with typos.
- ✅ **Security** — the ORM parameterizes queries for you, protecting against SQL injection.
- ✅ **Portability** — switch from SQLite to PostgreSQL later with (almost) zero code changes.
- ✅ **Readability** — `Book.query.all()` reads like English.

> ⚠️ An ORM doesn't *replace* SQL knowledge — it generates SQL under the hood. Understanding what SQL it produces makes you a much stronger developer.

### 2. Flask-SQLAlchemy

**SQLAlchemy** is the most popular Python ORM. **Flask-SQLAlchemy** is a small wrapper that integrates it neatly with Flask — it reads your Flask config, manages database sessions per request, and gives every model handy helpers like `Book.query`.

The two key config lines in [`main.py`](main.py):

```python
app.config['SQLALCHEMY_DATABASE_URI'] = 'sqlite:///books.db'
app.config['SQLALCHEMY_TRACK_MODIFICATIONS'] = False
```

- `SQLALCHEMY_DATABASE_URI` tells SQLAlchemy **where the database lives**. `sqlite:///books.db` means "a SQLite file named `books.db`" — Flask places it inside the `instance/` folder. If we moved to PostgreSQL one day, only this line would change (e.g. `postgresql://user:pass@localhost/books`).
- `SQLALCHEMY_TRACK_MODIFICATIONS = False` disables a legacy event system that adds overhead — always set this to `False`.

### 3. Why `extensions.py` Exists (Avoiding Circular Imports)

This is the "clean wiring" pattern demonstrated in class. Look at [`extensions.py`](extensions.py):

```python
from flask_sqlalchemy import SQLAlchemy
db = SQLAlchemy()
```

**Why not just create `db` inside `main.py`?** Because of a classic Python trap: the **circular import**.

- `models/book.py` needs `db` to define the model.
- `main.py` needs to import the models.
- If `db` lived in `main.py`, then `book.py` would import from `main.py`, while `main.py` imports from `book.py` → 💥 **circular import error**.

The fix: put `db` in a neutral third file that *both* can import from safely.

```
extensions.py  ←──imports──  models/book.py
      ↑
      └────────imports──────  main.py
```

Notice the two-step initialization in `main.py`:

```python
db = SQLAlchemy()      # in extensions.py — created WITHOUT an app
db.init_app(app)       # in main.py — bound to the app LATER
```

This "create now, bind later" pattern is called **deferred initialization**, and it's the standard way to structure Flask apps as they grow.

### 4. Models — Your Tables as Python Classes

Open [`models/book.py`](models/book.py):

```python
class Book(db.Model):
    __tablename__ = "books"

    id = db.Column(db.Integer, primary_key=True)
    title = db.Column(db.String(100), nullable=False)
    author = db.Column(db.String(100), nullable=False)
```

Line by line:

- **`class Book(db.Model)`** — inheriting from `db.Model` is what registers this class with SQLAlchemy as a database table.
- **`__tablename__ = "books"`** — explicitly names the table. (If omitted, SQLAlchemy guesses from the class name.)
- **`db.Column(...)`** — each attribute becomes a column. The first argument is the **type** (`Integer`, `String(100)`, etc.).
- **`primary_key=True`** — the unique identifier for each row. The database auto-increments it for us.
- **`nullable=False`** — the database itself will refuse a book without a title or author. Validation at the database level is your last line of defense.
- **`__repr__`** — purely for humans: when you print a `Book` in the shell you see `<Book Dune by Frank Herbert>` instead of `<Book object at 0x104f3d0>`. Great for debugging.

### 5. Migrations — Version Control for Your Database

Here's the problem migrations solve. Your **code** is version-controlled by git. But your **database schema** (the shape of your tables) changes over time too:

- Week 1: `books` has `id`, `title`, `author`.
- Week 3: you add a `year_published` column.
- Week 5: you add a whole new `authors` table.

How does your teammate's database — or the production database — get those same changes, **in the same order, without losing data**? That's what migrations do.

> 💡 **Think of migrations as git commits for your database schema.** Each migration file in `migrations/versions/` is one recorded change, with an `upgrade()` (apply the change) and a `downgrade()` (undo it).

Open the initial migration in [`migrations/versions/`](migrations/versions/) and you'll see exactly this:

```python
def upgrade():
    op.create_table('books',
        sa.Column('id', sa.Integer(), nullable=False),
        sa.Column('title', sa.String(length=100), nullable=False),
        sa.Column('author', sa.String(length=100), nullable=False),
        sa.PrimaryKeyConstraint('id')
    )

def downgrade():
    op.drop_table('books')
```

**Flask-Migrate** is a thin wrapper around **Alembic** (SQLAlchemy's migration engine). Its superpower: it **compares your models to the actual database** and auto-generates the migration script for you. You change the Python class; it figures out the SQL.

## 🚀 Getting Started (Setup)

Follow these steps **in order** after cloning the repo.

### 1. Clone and enter the project

```bash
git clone <repo-url>
cd app
```

### 2. Create and activate a virtual environment

A virtual environment keeps this project's packages isolated from your system Python.

```bash
python3 -m venv env
source env/bin/activate        # macOS / Linux
# env\Scripts\activate         # Windows (PowerShell: env\Scripts\Activate.ps1)
```

You'll know it worked when your prompt shows `(env)`.

### 3. Install the dependencies

```bash
pip install -r requirements.txt
```

This installs Flask, Flask-SQLAlchemy, Flask-Migrate, and everything they depend on — at the exact versions pinned in [`requirements.txt`](requirements.txt).

### 4. Create the database by applying migrations

The database file (`instance/books.db`) is **not** in the repo — you build it from the migration scripts:

```bash
export FLASK_APP=main.py       # tells the `flask` command where the app is
flask db upgrade
```

`flask db upgrade` runs every migration in `migrations/versions/` in order, creating `instance/books.db` with the `books` table. This is the payoff of migrations: **anyone who clones the repo can reconstruct the exact same database schema with one command.**

### 5. Run the server

```bash
python main.py
```

Visit http://127.0.0.1:5000/ — you should see the welcome message. 🎉

## 🔄 The Migration Workflow

This is the cycle you'll repeat every time the schema changes. Memorize it.

```
┌────────────────────┐     ┌──────────────────────┐     ┌────────────────────┐
│ 1. Edit the model  │ ──▶ │ 2. Generate migration │ ──▶ │ 3. Apply migration │
│   (models/book.py) │     │   flask db migrate    │     │   flask db upgrade │
└────────────────────┘     └──────────────────────┘     └────────────────────┘
```

### The three commands

| Command | What it does | When to run it |
|---|---|---|
| `flask db init` | Creates the `migrations/` folder | **Once ever, per project** (already done here — don't run it again) |
| `flask db migrate -m "message"` | **Detects** the difference between your models and the database, and **writes** a new migration script | Every time you change a model |
| `flask db upgrade` | **Applies** pending migration scripts to the database | After `migrate`, and after pulling teammates' migrations |

### Worked example: adding a `year_published` column

1. Edit `models/book.py`:

   ```python
   year_published = db.Column(db.Integer, nullable=True)
   ```

2. Generate the migration:

   ```bash
   flask db migrate -m "add year_published to books"
   ```

   ➡️ A new file appears in `migrations/versions/`. **Open it and read it** — check the generated `upgrade()` does what you expect.

3. Apply it:

   ```bash
   flask db upgrade
   ```

   ➡️ The `books` table now has the new column, and existing rows are untouched.

### Two crucial rules

- 🚫 `flask db migrate` **only writes the script** — it does NOT touch the database. Nothing changes until you `upgrade`. (Forgetting `upgrade` is the #1 beginner mistake.)
- ✅ **Commit migration files to git.** They're code. Your teammates need them to update *their* databases.

### Undoing a migration

```bash
flask db downgrade    # roll back the most recent migration
flask db history      # see all migrations
flask db current      # see which migration the DB is currently at
```

## 🐚 Playing With the ORM in the Flask Shell

The best way to *feel* what the ORM does. With your venv active:

```bash
flask shell
```

Then try:

```python
from extensions import db
from models.book import Book

# CREATE — add a row
book = Book(title="Things Fall Apart", author="Chinua Achebe")
db.session.add(book)      # stage it (like `git add`)
db.session.commit()       # save it  (like `git commit`)

# READ — query rows
Book.query.all()                          # every book
Book.query.get(1)                         # by primary key
Book.query.filter_by(author="Chinua Achebe").all()   # with a condition
Book.query.filter(Book.title.contains("Fall")).all() # fuzzy match

# UPDATE — change an object, then commit
book = Book.query.get(1)
book.title = "Things Fall Apart (Anniversary Edition)"
db.session.commit()

# DELETE
book = Book.query.get(1)
db.session.delete(book)
db.session.commit()
```

> 💡 **`db.session` is a staging area**, just like git's. `add()` / `delete()` stage changes; nothing hits the database until `commit()`. If something goes wrong, `db.session.rollback()` discards the staged changes.

## 💪 Your Turn: The Exercise

Look closely at the routes in [`main.py`](main.py). They still reference the **old in-memory `books` list** from before we wired up the database... but that list is gone. The routes are broken **on purpose** — fixing them is your assignment.

**Refactor every route to use the ORM instead of the list.** For example, `GET /books` becomes:

```python
@app.route('/books')
def get_books():
    books = Book.query.all()
    return jsonify([{"id": b.id, "title": b.title, "author": b.author} for b in books])
```

Your checklist:

- [ ] `GET /books` — return all books from the database
- [ ] `GET /books/<id>` — use `Book.query.get(...)`, return 404 if missing

- [ ] `PUT /books/<id>` — fetch, modify attributes, `commit()`
- [ ] `DELETE /books/<id>` — fetch, `db.session.delete()`, `commit()`

## 🛠 Troubleshooting

| Problem | Likely cause & fix |
|---|---|
| `Error: Could not locate a Flask application` | You forgot `export FLASK_APP=main.py` |
| `No changes in schema detected` when running `migrate` | Your model isn't imported in `main.py` — Flask-Migrate can only see models that get imported. Also check you actually changed the model. |
| `no such table: books` | You never ran `flask db upgrade` — the migration exists but was never applied |
| Changes to a model not showing in the DB | You ran `migrate` but forgot `upgrade` |
| `ModuleNotFoundError: No module named 'flask'` | Your venv isn't activated (`source env/bin/activate`) |
| Weird import errors mentioning `main` and `book` | Circular import — make sure models import `db` from `extensions`, never from `main` |
| Database in a strange state during practice | It's just SQLite — delete `instance/books.db`, run `flask db upgrade`, start fresh |
